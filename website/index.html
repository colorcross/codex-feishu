Session-aware bridge for disciplined engineering workflows

把飞书变成 Codex 的工作入口。

Codex Feishu 让飞书消息直接进入可续接的 Codex 会话。项目可路由，回复可追踪，运行可运维。它不是一次性 bot，而是一层能长期运行的工程桥接器。

What this keeps stable
  • 同一个聊天入口，可以路由到多个仓库目录
  • 同一个项目，可以持续续接同一条 Codex 会话
  • 运行状态、审计、指标与取消能力保留在本地控制面
Transport long-connection + webhook

Routing Selection / Session / Queue 分离

避免聊天窗口状态碰撞，保证项目切换可解释。

Operations Cancel / Timeout / Recovery

不是只会转发消息，还能管理运行生命周期。

Why this exists

飞书负责沟通，Codex 负责执行，桥接器负责把上下文和控制权保住。

Route by project

用 `/project <alias>` 明确当前仓库，而不是让每条消息都重新解释上下文。

Resume by session

新会话走 `codex exec`，续会话走 `codex exec resume`，让多轮任务真正延续。

Operate with confidence

幂等去重、原生 reply、审计日志、指标和运行态保护，保证它能长期工作而不是偶尔成功。

Deployment posture

两种接入姿态，分别服务本机效率和团队共享。

Local-first

long-connection

不依赖公网回调地址，最适合个人开发、内网环境和快速联调。你在飞书里发消息，桥接器在本机直接收、直接路由、直接执行。

  • 接入快
  • 部署简单
  • 适合长期个人工作流
Shared service

webhook

适合团队共享服务、卡片回调、反向代理和标准生产接入。把飞书消息入口接到一个稳定、可观测、可运维的服务上。

  • 支持卡片回调
  • 便于 HTTPS 与代理接入
  • 适合正式服务化部署
Day-2 operations

真正的价值，不是第一次跑起来，而是之后每一天都能被管理。

Operational surface
Runtime discipline

支持取消、超时、stale run 恢复和实例锁，避免一条消息把长期运行环境拖进不可控状态。

Message discipline

过滤机器人自激回环，保留消息幂等去重，并优先使用飞书原生 reply 维持对话上下文。

Observability

审计日志、本地 run state、Prometheus 指标和 Grafana / Alertmanager 示例一起提供。

Documentation
